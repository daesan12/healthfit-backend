# HealthFit Backend API Specification

이 문서는 현재 Django 백엔드 구현을 기준으로 작성된 프론트엔드 연동 계약이다.

- Base URL: `http://127.0.0.1:8000/api/v1`
- 인증 방식: JWT Bearer
- 최종 갱신: 2026-06-20
- 현재 구현된 method/path: 67개
- 향후 구현 예정 method/path: 6개 (`Planned / Not Implemented Yet APIs`에 별도 표기)

## 1. 공통 규칙

인증이 필요한 요청:

```http
Authorization: Bearer {access_token}
Content-Type: application/json
```

성공 응답:

```json
{
  "success": true,
  "message": "요청별 성공 메시지",
  "data": {}
}
```

실패 응답:

```json
{
  "success": false,
  "message": "요청별 실패 메시지",
  "errors": {
    "field": ["오류 메시지"]
  }
}
```

프론트엔드는 Axios 원본 응답을 직접 사용할 때 `response.data.data`에서 실제 데이터를 읽는다.
`frontend/src/api/client.js`의 `unwrapResponse()`를 거친 함수는 이미 내부 `data`를 반환하므로 반환값을 그대로 사용한다.

## 3. Auth

### POST `/api/v1/auth/signup/`

Request:

```json
{"username":"user01","email":"user01@example.com","password":"password123!"}
```

Response:

```json
{"success":true,"message":"회원가입이 완료되었습니다.","data":{"id":1,"username":"user01","email":"user01@example.com","created_at":"2026-06-20T10:00:00Z"}}
```

### POST `/api/v1/auth/login/`

`login_id`에는 username 또는 email을 보낸다.

```json
{"login_id":"user01","password":"password123!"}
```

```json
{"success":true,"message":"로그인에 성공했습니다.","data":{"access":"jwt_access_token","refresh":"jwt_refresh_token","user":{"id":1,"username":"user01","email":"user01@example.com"}}}
```

### POST `/api/v1/auth/logout/`

```json
{"refresh":"jwt_refresh_token"}
```

```json
{"success":true,"message":"로그아웃되었습니다.","data":null}
```

### GET `/api/v1/auth/me/`

```json
{"success":true,"message":"내 정보 조회 성공","data":{"id":1,"username":"user01","email":"user01@example.com","has_profile":true}}
```

PUT은 부분 수정이 아니라 전체 필드를 요구한다.

```json
{
  "gender":"male",
  "age":27,
  "height":181.0,
  "weight":100.0,
  "body_type":"overweight",
  "activity_level":"low",
  "workout_goal":"fat_loss",
  "workout_experience":"beginner"
}
```

GET/PUT data 예시:

```json
{"id":1,"gender":"male","age":27,"height":181.0,"weight":100.0,"body_type":"overweight","activity_level":"low","workout_goal":"fat_loss","workout_experience":"beginner"}
```

### GET `/api/v1/profiles/me/calorie-target/`

```json
{"success":true,"message":"권장 칼로리 계산이 완료되었습니다.","data":{"recommended_calories":2200,"carbohydrate_ratio":50,"protein_ratio":30,"fat_ratio":20,"recommended_carbohydrate":275.0,"recommended_protein":165.0,"recommended_fat":48.9}}
```

## 5. Body Record

GET query: `start_date`, `end_date` (`YYYY-MM-DD`, 선택).

POST:

```json
{"record_date":"2026-06-20","weight":70.5,"body_fat_percentage":18.2,"skeletal_muscle_mass":32.4}
```

PATCH는 위 필드 일부만 보낸다. `bmi`는 프로필 키와 기록 체중으로 서버가 계산하며 DB 입력 필드가 아니다.

Record data:

```json
{"id":1,"record_date":"2026-06-20","weight":70.5,"body_fat_percentage":18.2,"skeletal_muscle_mass":32.4,"bmi":21.52,"created_at":"2026-06-20T10:00:00Z"}
```

GET 목록의 `data`는 Record 배열이다. DELETE 성공의 `data`는 `null`이다.

## 6. Food

GET query: `search`, `category`, `mine=true`.

- 비로그인: 기본 음식(`user_id=null`)만 조회
- 로그인: 기본 음식과 내 음식 조회
- `mine=true`: 내 음식만 조회
- 기본 음식과 다른 사용자의 음식은 수정/삭제 불가

POST:

```json
{"name":"닭가슴살 구이","category":"구이류","calories":165,"carbohydrate":0,"protein":31,"fat":3.6}
```

Food data:

```json
{"id":1,"name":"닭가슴살 구이","category":"구이류","calories":165.0,"carbohydrate":0.0,"protein":31.0,"fat":3.6,"user_id":null}
```

PATCH는 Food 필드 일부를 보내며 DELETE의 `data`는 `null`이다.

## 9. Exercise

GET query: `search`, `body_part`, `equipment`, `target_muscle`.

- 상세 URL의 `{pk}`와 관계 필드의 `exercise_id`는 숫자 Django PK이다.
- Exercise data 안의 문자열 `exercise_id`는 ExerciseDB 원본 ID 또는 `custom-...` ID이다.

POST 사용자 운동:

```json
{"name":"개인 스트레칭","gif_url":null,"body_parts":["어깨"],"equipments":["맨몸"],"target_muscles":["삼각근"],"secondary_muscles":[],"instructions":["천천히 어깨를 돌립니다."]}
```

Exercise data:

```json
{"id":1,"exercise_id":"01qpYSe","name":"엎드린 강아지 자세","gif_url":"https://static.exercisedb.dev/media/01qpYSe.gif","body_parts":["등"],"equipments":["맨몸"],"target_muscles":["척추기립근"],"secondary_muscles":["어깨"],"instructions":["1단계: 자세를 준비합니다."],"user_id":null,"created_at":"2026-06-20T10:00:00Z","updated_at":"2026-06-20T10:00:00Z"}
```

비로그인은 기본 운동만, 로그인 사용자는 기본 운동과 내 운동만 조회한다. 내 사용자 운동만 PATCH/DELETE할 수 있다.

## 10. Workout Routine and Routine Item

POST `/workout-routines/`은 항목을 받지 않고 루틴만 생성한다.

```json
{"name":"월수금 전신 루틴","description":"초보자용 루틴"}
```

Routine data:

```json
{
  "id":1,
  "name":"월수금 전신 루틴",
  "description":"초보자용 루틴",
  "items":[{"id":1,"exercise":{"id":2,"exercise_id":"05Cf2v8","name":"불가능한 딥스","gif_url":null,"body_parts":["상완"],"equipments":["맨몸"],"target_muscles":["삼두근"],"secondary_muscles":[],"instructions":[],"user_id":null,"created_at":"2026-06-20T10:00:00Z","updated_at":"2026-06-20T10:00:00Z"},"order":1,"sets":3,"reps":12,"weight":20.0,"rest_seconds":60}],
  "created_at":"2026-06-20T10:00:00Z",
  "updated_at":"2026-06-20T10:00:00Z"
}
```

POST `/workout-routines/{routine_id}/items/`:

```json
{"exercise_id":2,"order":1,"sets":3,"reps":12,"weight":20,"rest_seconds":60}
```

`exercise_id`는 숫자 Exercise PK이다. PATCH `/routine-items/{routine_item_id}/`은 위 필드 일부를 받는다.
RoutineItem 응답은 숫자 `exercise_id`를 별도 필드로 반환하지 않고 중첩 `exercise` 객체를 반환한다.

루틴 생성/조회/수정/삭제 및 항목 변경은 모두 현재 로그인 사용자의 루틴으로 제한된다.

## 11. Workout Log

GET query: `date`, `start_date`, `end_date`, `workout_id`, `routine_id`. 기존 `exercise_id` query도 `workout_id` 별칭으로 허용한다.

POST:

```json
{"workout_id":1,"routine_id":1,"workout_date":"2026-06-20","workout_time":30,"memo":"사용자 메모","sets":[{"set_order":1,"weight_kg":60,"repetition":10,"duration_seconds":null,"rpe":8,"is_warmup":false},{"set_order":2,"weight_kg":60,"repetition":10,"duration_seconds":null,"rpe":8.5,"is_warmup":false},{"set_order":3,"weight_kg":60,"repetition":8,"duration_seconds":null,"rpe":9,"is_warmup":false}]}
```

`workout_id`는 숫자 Exercise PK이며 `routine_id`는 생략하거나 `null`로 보낼 수 있다. 기존 프론트 호환을 위해 `exercise_id`, `set_count`, `repetition`만 보내는 요청도 허용하며, 이 경우 백엔드가 동일 반복 수의 세트를 생성한다. 새 연동은 `workout_id`와 `sets`를 사용한다.

WorkoutLog data:

```json
{"id":1,"log_id":1,"workout_id":1,"exercise_id":1,"exercise":{"id":1,"exercise_id":"01qpYSe","name":"Bench Press"},"routine_id":1,"workout_date":"2026-06-20","workout_time":30,"set_count":3,"repetition":10,"memo":"사용자 메모","sets":[{"id":1,"set_order":1,"weight_kg":60,"repetition":10,"duration_seconds":null,"rpe":8,"is_warmup":false}],"created_at":"2026-06-20T10:00:00Z","updated_at":"2026-06-20T10:00:00Z"}
```

`weight_kg`는 맨몸 운동에서 null, `repetition`은 시간 기반 운동에서 null일 수 있으며 이 경우 `duration_seconds`가 필요하다. `rpe`는 1~10의 선택값이다. PATCH의 `sets`는 기존 세트를 전체 교체하며, 로그 삭제 시 세트도 함께 삭제된다. 본인의 기록만 조회/수정/삭제할 수 있다.

## 14. AI Diet and Workout

모든 AI API는 JWT 인증이 필요하다. 실제 GMS 키는 `backend/.env`의 `GMS_KEY`에만 저장한다. 모델명은 `gemini-2.5-flash-lite`, endpoint는 SSAFY GMS 프록시 `https://gms.ssafy.io/gmsapi/generativelanguage.googleapis.com/v1beta`를 코드 기본값으로 사용하므로 다른 환경변수는 필요하지 않다.

GMS를 호출하는 평가·추천 API에서 키가 없으면 503과 다음 응답을 반환한다.

```json
{"success":false,"message":"AI API key is not configured.","errors":{"GMS_KEY":["GMS_KEY is missing."]}}
```

### POST `/api/v1/ai/diet/evaluations/`

내 프로필과 해당 날짜의 Meal/MealItem을 조회해 목표 영양량과 실제 섭취량을 GMS로 평가하고 `DietFeedback`에 저장한다. 프로필 또는 식단 기록이 없으면 400을 반환한다. 기존 명세 호환을 위해 `date`도 같은 의미로 허용하지만 프론트 신규 연동은 `target_date`를 사용한다.

```json
{"target_date":"2026-06-20"}
```

```json
{"success":true,"message":"AI 식단 평가가 완료되었습니다.","data":{"feedback_id":1,"date":"2026-06-20","score":78,"total_calories":1950,"total_carbohydrate":210,"total_protein":165,"total_fat":48,"strengths":["단백질 섭취가 적절합니다."],"improvements":["탄수화물을 조금 더 보충하면 좋습니다."],"recommended_actions":["저녁에 현미밥을 소량 추가하세요."],"feedback":"전체 칼로리는 목표보다 약간 부족하지만 단백질 섭취는 양호합니다."}}
```

기존 응답 필드 `id`, `target_date`, `summary`, `good_points`, `improvement_points`, `recommendation`, `result_data`도 호환을 위해 함께 반환한다. 총 영양량은 AI가 아니라 저장된 Meal 합계를 사용한다.

### POST `/api/v1/ai/diet/recommendations/`

`scope`는 `meal`(기본값), `day`, `remaining` 중 하나다. `food_source`는 다음 중 하나다.

- `all`: 전체 음식 DB, 기본 음식과 내 사용자 음식 사용
- `my_fridge`: 나의 냉장고, 내 사용자 음식만 사용
- `free`: DB 밖 음식 허용. 각 항목의 100g 영양 스냅샷으로 Meal/SavedMeal/SavedMealPlan 저장 가능

```json
{"scope":"meal","message":"맵지 않은 한식 고단백 저녁을 추천해줘.","target_date":"2026-06-21","food_source":"all","meal_order":1,"meal_label":"저녁"}
```

하루 5끼 예시:

```json
{"scope":"day","message":"새우를 제외한 한식 고단백 감량 식단","target_date":"2026-06-21","food_source":"all","meal_count":5}
```

남은 영양량 기준 한 끼 예시:

```json
{"scope":"remaining","message":"남은 칼로리에 맞는 간단한 저녁","target_date":"2026-06-21","food_source":"all","meal_order":4,"meal_label":"저녁"}
```

`day`의 `meal_count`는 1~6이며 생략 시 3이다. JSON body의 `scope`, `meal_count`, `meal_order`, `meal_label`이 자연어 message와 GMS 조건 분석보다 우선한다. GMS가 다른 끼니 수를 반환하면 백엔드는 정확한 끼니 수를 명시해 한 번 교정 재요청하고, 두 번째도 틀리면 502 검증 오류를 반환한다. `meal`과 `remaining`은 `meal_order`, `meal_label`을 사용한다. `meal_type`은 기존 연동 호환용 선택 필드다. `exclude_food_ids`, `preference`도 계속 허용한다.

DB 기반 item은 `food_id`, `name`, `amount`, `role`과 백엔드 재계산 영양값을 반환한다. `free` item은 다음 구조이며 `food_id`는 null이다.

```json
{"food_id":null,"ai_food_key":"free_salmon_001","name":"연어구이","amount":150,"role":"protein","nutrition_per_100g":{"calories":208,"carbohydrate":0,"protein":20,"fat":13},"calories":312,"carbohydrate":0,"protein":30,"fat":19.5}
```

응답의 `data.meals`는 식사 배열이고 각 원소에는 `meal_order`, `meal_label`, `target_calories`, `items`, 영양 합계가 있다. `day`는 `daily_target`, `daily_totals`를, `remaining`은 `already_eaten`, `remaining_target`, `meal`을 추가로 반환한다. 한 끼 기존 연동을 위해 첫 식사의 `items`와 영양 합계도 data 최상위에 함께 반환한다.

### GET `/api/v1/ai/diet/recommendations/{recommendation_id}/`

본인 추천만 조회한다. `data.content`에 생성 당시 전체 추천 결과가 들어 있으며 다른 사용자의 추천은 404를 반환한다.

### POST `/api/v1/ai/workout/recommendations/`

`exercise_source`:

- `all`: 전체 운동 DB, 기본 운동과 내 사용자 운동 사용
- `my_exercises`: 내 운동만 사용

```json
{"message":"초보자용 가슴 루틴 추천해줘","target_body_part":"chest","available_time":60,"exercise_source":"all"}
```

기존 planned 명세의 `weekly_frequency`, `preference`도 받을 수 있다. 응답과 저장 요청의 `exercise_id`는 외부 문자열이 아니라 숫자 Exercise DB PK이다.

```json
{"success":true,"message":"AI 운동 루틴 추천이 완료되었습니다.","data":{"recommendation_id":3,"type":"workout","exercise_source":"all","save_available":true,"title":"초보자용 가슴 루틴","description":"초보자가 수행하기 쉬운 운동으로 구성했습니다.","items":[{"exercise_id":1,"exercise_name":"푸쉬업","order":1,"sets":3,"reps":12,"weight":0,"rest_seconds":60}],"actions":[{"type":"save_routine","label":"내 루틴에 저장"}]}}
```

### POST `/api/v1/ai/workout/progression/`

특정 Exercise의 최근 세트별 기록과 target/secondary muscle의 마지막 훈련일을 이용해 다음 중량·반복·세트·휴식 시간을 추천한다. `workout_id`는 숫자 Exercise PK다. `target_date` 생략 시 오늘, `goal` 생략 시 프로필의 `workout_goal`을 사용한다.

```json
{"workout_id":12,"target_date":"2026-06-24","goal":"muscle_gain","message":"다음 벤치프레스 목표를 추천해줘"}
```

```json
{"success":true,"message":"운동 진행 추천이 완료되었습니다.","data":{"workout_id":12,"workout_name":"Bench Press","target_date":"2026-06-24","decision":"maintain","recommendation":{"set_count":3,"repetition":8,"weight_kg":62.5,"rest_seconds":120},"recovery_summary":[{"muscle":"pectorals","last_trained_date":"2026-06-21","days_since":3,"status":"recovered"},{"muscle":"delts","last_trained_date":"2026-06-23","days_since":1,"status":"not_fully_recovered"}],"history_summary":[{"date":"2026-06-21","sets":[{"set_order":1,"weight_kg":62.5,"repetition":8,"duration_seconds":null,"rpe":8,"is_warmup":false}],"total_volume":1500}],"reason":"최근 반복 수와 보조근육 회복 상태를 고려해 현재 중량을 유지합니다.","safety_note":"체감 난이도에 따라 중량, 반복 수, 세트 수를 직접 조절하세요."}}
```

`decision`은 `increase`, `maintain`, `decrease`, `deload`, `bodyweight_progression`, `insufficient_history` 중 하나다. 백엔드가 먼저 보수적인 결정을 만들고 GMS는 설명과 안전 범위 안의 미세 조정만 담당한다. 관련 근육을 0~1일 전에 훈련했다면 중량 증가는 허용하지 않는다. GMS 호출 실패·잘못된 JSON·과도한 값은 백엔드 예비 추천으로 fallback한다. 최근 같은 운동은 최대 5개 로그를 사용하며 memo는 GMS payload와 진행 판단에서 완전히 제외한다.

### AI 추천 저장

Meal 저장:

```http
POST /api/v1/ai/recommendations/{recommendation_id}/save-meal/
```

```json
{"meal_type":"dinner","intake_date":"2026-06-20"}
```

통합 식단 저장 경로다. `title`과 `description`은 추천 생성이 아니라 저장 시 전달한다.

```http
POST /api/v1/ai/diet/recommendations/{recommendation_id}/save/
```

```json
{"save_target":"meal_plan","title":"나의 하루 감량 식단","description":"한식, 새우 제외"}
```

`save_target`은 `meals`, `saved_meal`, `meal_plan`, `both` 중 하나다. 한 끼 추천은 `meals` 또는 `saved_meal`, 하루 추천은 `meals` 또는 `meal_plan`을 사용한다. `both`는 한 끼에서 Meal+SavedMeal, 하루에서 Meals+SavedMealPlan을 만든다. 기존 `{ "name": "...", "description": "..." }` 요청은 `saved_meal` 저장으로 호환된다. `free`도 FoodSnapshot을 통해 모든 저장 대상에 저장할 수 있다.

### POST `/api/v1/ai/diet/recommendations/{recommendation_id}/replace/`

원본을 수정하지 않고 `parent_recommendation`으로 연결된 새 추천을 만든다.

```json
{"meal_order":2,"replace_food_id":22,"message":"닭가슴살 대신 두부나 생선으로 바꿔줘"}
```

free 모드는 `replace_food_id` 대신 `replace_ai_food_key`를 사용한다.

```json
{"meal_order":2,"replace_ai_food_key":"free_chicken_001","message":"두부나 생선으로 바꿔줘"}
```

### POST `/api/v1/ai/diet/recommendations/{recommendation_id}/reroll/`

원래 scope, 날짜, food_source, 끼니 수를 유지하고 추가 조건으로 새 추천을 만든다.

```json
{"message":"닭가슴살은 제외하고 조리가 더 쉬운 음식으로 다시 추천해줘"}
```

WorkoutRoutine 저장:

```http
POST /api/v1/ai/recommendations/{recommendation_id}/save-routine/
```

```json
{"name":"AI 추천 가슴 루틴","description":"초보자용 가슴 운동 루틴"}
```

세 저장 API는 추천 소유권과 종류를 확인한 뒤 기존 Meal, SavedMeal, RoutineItem serializer로 실제 데이터를 생성한다. 다른 사용자 음식·운동 ID는 저장할 수 없다.

## 15. 프론트엔드 연동 주의사항

1. `/api/v1/workouts/`는 존재하지 않는다. 운동 목록은 `/api/v1/exercises/`를 사용한다.
2. 식단 대시보드는 `/api/v1/meals/dashboard/`이다.
3. WorkoutRoutine 생성 body는 `name`, `description`만 사용한다. 항목은 RoutineItem API로 별도 추가한다.
4. RoutineItem은 `exercise_id`, 새 WorkoutLog/AI progression은 `workout_id`를 사용하며 모두 숫자 Exercise PK이다. WorkoutLog의 기존 `exercise_id`도 호환된다.
5. Exercise 응답의 문자열 `exercise_id`는 외부 ExerciseDB 식별자이므로 숫자 PK인 `id`와 구분한다.
6. SavedMeal 제목 필드는 `title`이 아니라 `name`이다.
7. Post 목록의 `data`는 배열이 아니라 `{count, page, page_size, results}` 객체이다.
8. 공개 GET이라도 유효하지 않은 Bearer token을 보내면 인증 오류가 발생할 수 있다.
9. DELETE 성공 응답은 공통 envelope의 `data: null`이다.
10. 오류 HTTP status는 주로 400, 401, 404이며 오류 본문은 `error.response.data`에서 읽는다.

11. AI 생성·교체·재추천 API는 `GMS_KEY`가 필요하다. `free` 식단도 통합 저장 API로 저장할 수 있다.

## 16. Planned / Not Implemented Yet APIs

> 프론트 연동 시에는 Implemented APIs 섹션만 우선 사용한다. Planned / Not Implemented Yet APIs는 이후 구현 예정이며, 현재 백엔드에서는 호출하면 404가 발생할 수 있다.

아래 6개 method/path는 기존 프로젝트 명세에 존재하지만 현재 백엔드에는 구현되지 않았다. 구현 완료 전까지 프론트 운영 코드에서 호출하지 않는다.

| 그룹 | Method | Endpoint | 인증 | 목적 | 상태 |
|---|---|---|---|---|---|

### Planned Meal Item

POST `/api/v1/meals/{meal_id}/items/` 예상 request:

```json
{"food_id":3,"amount":200}
```

PATCH `/api/v1/meal-items/{meal_item_id}/` 예상 request:

```json
{"amount":180}
```

예상 성공 data는 `MealItem`이며 추가/수정/삭제 후 Meal 총 영양성분을 다시 계산한다. DELETE의 예상 data는 `null`이다.

### Planned Diet Feedback

GET `/api/v1/diet-feedbacks/` 예상 query: `date`, `start_date`, `end_date`. 예상 response data는 저장된 `DietFeedback` 객체 배열이다.

### Planned AI Chat

POST `/api/v1/ai/chats/` 예상 request:

```json
{"question":"다이어트 중 단백질은 얼마나 먹어야 하나요?"}
```

예상 response data:

```json
{"id":1,"question":"다이어트 중 단백질은 얼마나 먹어야 하나요?","answer":"AI 답변","created_at":"2026-06-20T10:00:00Z"}
```

GET `/api/v1/ai/chats/`의 예상 response data는 위 AI Chat 객체 배열이다.
